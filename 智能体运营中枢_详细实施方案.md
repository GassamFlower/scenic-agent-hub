# 「门神文化景区・LangGraph 多智能体运营中枢」落地实施方案

**文档说明**：本文档基于《门神文化景区・LangGraph 多智能体运营中枢方案》，作为项目的**执行级实施指南**，详细阐述从业务梳理、技术选型细节到开发落地的具体路线、时间节点与风险控制，旨在为研发团队、运营团队与管理层提供清晰的行动手册。

---

## 一、 业务数据流转与触点落地设计

为了确保线上线下业务不割裂，我们需要对用户的核心触点及数据流转进行详细设计。

### 1. 全域触点集成方案
| 触点来源 | 接入方式 | 核心承载业务 | 预期频率/并发 |
| :--- | :--- | :--- | :--- |
| **微信小程序** | WebSocket/长连接 + REST API | 票务核销、研学预约、商城导购、AI专属客服 | 高，占总流量 60% |
| **微信公众号** | 微信开放平台消息接口 | 消息推送、长图文营销、简单被动回复 | 中，占总流量 20% |
| **抖音矩阵** | Coze 插件 / 抖音开放平台API | 评论区自动回复、意向线索留资 | 中，占总流量 15% |
| **Web 管理后台** | React/Vue SPA + FastAPI | 运营数据看板、人工客服接管、大模型配置 | 低，内部使用 |

### 2. 核心数据模型 (Data Schema)
*   **User (用户态)**：`user_id` (OpenID/UnionID), `tags` (如：带娃、文创偏好), `loyalty_points` (积分)。
*   **Session (会话态)**：`session_id`, `context` (近10轮对话历史), `current_intent` (当前意图)。
*   **Order (交易态)**：`order_id`, `status` (已支付/已核销/退款), `type` (门票/研学/文创)。

## 二、 核心技术栈详细使用规范

### 1. LangGraph：中枢图结构 (Graph Schema) 设计
LangGraph 并非简单的链式调用，我们需要定义一个状态图 (State Graph)。

*   **State 定义 (`AgentState`)**：
    ```python
    from typing import TypedDict, Annotated, Sequence
    from langchain_core.messages import BaseMessage
    import operator

    class AgentState(TypedDict):
        messages: Annotated[Sequence[BaseMessage], operator.add]
        user_id: str
        is_authenticated: bool
        has_ticket: bool
        current_intent: str # [客服, 研学, 票务, 电商]
        requires_human: bool # 是否触发人工接管
    ```
*   **节点 (Nodes) 与 边 (Edges) 路由**：
    *   `Gateway_Node` (大模型意图识别)：分析用户最新消息，更新 `current_intent`。
    *   `Router_Edge`：根据 `current_intent` 分发到对应的 CrewAI 智能体。
    *   `Human_in_loop_Edge`：当置信度低于 60% 或触发退款等敏感词时，路由至 `Human_Fallback_Node`。

### 2. CrewAI：智能体角色卡设定 (Role Prompting)
在 CrewAI 中，需要为每一个 Agent 提供极其精确的设定，避免其偏离业务。
*   **示例：票务智能体 (Ticketing Agent)**
    *   **Role**: 门神文化景区资深票务主管
    *   **Goal**: 解决游客关于门票、预约、一票多用规则的疑问，并引导完成购票。
    *   **Backstory**: 你在景区工作了5年，熟知所有的退改签规则和周边合作商户的打折活动。你必须严格调用票务API查询真实库存，绝对不能编造门票价格。
    *   **Tools**: `@tool("query_ticket_price")`, `@tool("check_inventory")`, `@tool("generate_payment_link")`

### 3. Dify：RAG 知识库构建标准
避免大模型“胡说八道”（幻觉）的关键在于 Dify 知识库的实施质量。
*   **数据清洗**：将《景区 QA 手册》、《门神文化大纲》清洗为 Markdown 格式，去除冗余修饰词。
*   **分块策略 (Chunking)**：采用按标题/段落分块，Chunk Token 限制在 500 左右，Overlap 设为 50，保证上下文连贯。
*   **检索策略**：采用**混合检索 (Hybrid Search)**，即 `全文检索 (BM25)` + `向量检索 (Vector)`，并开启 `Rerank (重排)`，确保专业词汇（如特定门神名字）的准确召回。

### 4. OpenClaw / API Tool Calling：执行层对接
*   **票务系统对接**：通过 FastAPI 封装景区原有老旧票务系统的 SOAP/HTTP 接口，提供标准的 RESTful 工具给 AI 调用。
*   **微信支付**：封装生成支付参数的 Tool，AI 仅返回支付参数组合，最终由前端小程序拉起收银台，**绝对禁止 AI 直接操作资金划转**。

## 三、 详细实施路线图 (里程碑计划)

项目预计分为 4 个阶段，总周期约 **10-12 周**。

### 阶段一：基础设施与知识库搭建（MVP阶段） —— 第 1-3 周
*   **目标**：跑通主干流程，智能客服具备基本问答能力。
*   **任务清单**：
    1.  梳理 100+ 个高频游客 QA 及 5 万字门神文化资料。
    2.  在 Dify 中创建向量知识库，完成测试调优（回答准确率需达 90%）。
    3.  搭建 FastAPI 后端，实现微信小程序的文本消息接收与响应（目前用的时微信小程序自带云服务，可以对接dify）。
    4.  **产出物**：知识库系统、可文字对话的小程序测试版（仅客服意图）。

### 阶段二：LangGraph中枢与核心业务流打通 —— 第 4-6 周
*   **目标**：引入状态管理，票务与研学业务闭环。
*   **任务清单**：
    1.  开发 LangGraph 的核心路由节点与记忆模块（采用 Redis/PostgreSQL 存储 Checkpoint）。
    2.  配置 CrewAI 的【票务智能体】与【研学智能体】。
    3.  开发 API Tools（票务查询、生成订单、生成小程序卡片）。
    4.  **产出物**：支持复杂上下文对话、能在对话中直接推送购票/研学卡片的小程序正式版。

### 阶段三：内容营销与多渠道扩展 —— 第 7-9 周
*   **目标**：释放运营人力，全域流量覆盖。
*   **任务清单**：
    1.  配置【内容运营智能体】与【文创电商导购智能体】。
    2.  对接 OpenClaw / QClaw，实现生成排版好的长图文并存入草稿箱。
    3.  利用 Coze 搭建抖音矩阵的评论区监控与自动私信引流流。
    4.  **产出物**：内容自动化生产工作流、抖音 AI 客服、电商导购服务。

### 阶段四：监控管理与平稳交付 —— 第 10-12 周
*   **目标**：数据可视化，完善异常处理机制。
*   **任务清单**：
    1.  开发 React/Vue 运营管理后台看板。
    2.  配置【管理后台智能体】，通过自然语言查询系统数据（Text2SQL）。
    3.  完善 LangGraph 的 `Human-in-the-loop`（人在回路），实现人工客服一键强插接管。
    4.  压力测试与安全渗透测试。
    5.  **产出物**：完整交付整个多智能体运营中枢、管理后台、实施与运维手册。

## 四、 风险评估与应急预案

| 风险类别 | 具体风险描述 | 应对策略 / 应急预案 |
| :--- | :--- | :--- |
| **技术风险** | 大模型产生严重幻觉，报出错误票价 | 1. 票务数据强制从 Tool API 获取，不依赖大模型记忆。<br>2. 最终支付金额以系统收银台为准，前端做二次强校验。 |
| **合规风险** | 用户输入违规/敏感词汇导致账号封禁 | 1. 接入大厂（如阿里云/腾讯云）文本安全审核API。<br>2. 在 LangGraph 入口节点设置拦截网关，触发则直接回复标准致歉话术。 |
| **性能风险** | 节假日并发激增，大模型 API 限流或响应慢 | 1. 设置长连接 loading 状态（“正在为您翻阅古籍...”）。<br>2. Redis 缓存高频完全相同的提问（如“今天几点关门”）。<br>3. 准备降级方案，切换回传统规则树/人工客服。 |
| **业务风险** | 老票务系统接口不稳定，AI 调用超时 | 为所有 API Tool 设置严格的 Timeout（如 3秒）。超时则触发 Exception，AI 转述：“抱歉，系统太拥挤，请点击下方按钮直接前往票务大厅查看”。 |

## 五、 团队配置建议

为了保障该实施方案顺利落地，建议配备以下项目团队：
*   **AI 架构师 / 资深后端 (1人)**：负责 LangGraph 状态图设计、FastAPI 核心网关、Prompt 工程。
*   **业务后端开发 (1-2人)**：负责编写 Tool API、对接老票务系统、数据库操作。
*   **前端开发 (1人)**：负责微信小程序对话界面改造、自定义组件卡片（如购票卡片）渲染。
*   **产品运营 / 业务专家 (1人)**：负责整理 Dify 知识库语料、设定 CrewAI 角色性格、验收最终对话效果。
