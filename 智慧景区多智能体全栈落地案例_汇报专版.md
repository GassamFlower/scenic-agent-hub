## 1. 💡 项目背景与核心业务价值

### 1.1 行业痛点

传统景区的数字化服务通常是“割裂”的：游客查攻略在一个App，买门票在一个小程序，客服咨询又需要人工介入，且常常回复滞后。同时，景区自身的数据资产（如导游词、历史文化、运营规范）缺乏盘活，营销高度依赖人工。

### 1.2 我们的解决方案

本项目旨在打造一个\*\*“能听懂、会思考、能办事”的景区超级大脑\*\*。
打通了从“前端微信小程序（购票/导览）”到“后端多智能体协作（LangGraph+CrewAI）”，再到“本地私有化企业知识库（Dify）”与“工具执行层（OpenClaw）”的全链路。

### 1.3 💰 核心业务价值 (商业收益)

- **极致的游客体验**：7x24小时的“金牌数字导游”，从路线规划到一键购票，全部在自然语言对话中完成。
- **显著的降本增效**：拦截 80% 的常规客服咨询，自动生成每日营销文案，极大降低人力成本。
- **数据资产安全可控**：核心系统（Dify、OpenClaw）全部采用**本地私有化部署**，景区敏感运营数据不出域。

## 3. ⚙️ 技术架构与底层框架 (硬核技术解析)

本项目采用目前最前沿的 `Agentic Workflow` (智能体工作流) 架构，彻底告别单体大模型的局限。

### 3.1 核心架构图 (LangGraph + CrewAI 协同)

```Markdown
graph TD
    User[微信小程序 / 用户请求] -->|HTTP/WS| API(FastAPI 网关)
    API --> Router{LangGraph 中枢路由}
    
    subgraph 🧠 AI 大脑层 (LangGraph + CrewAI)
        Router -->|状态与记忆管理| Memory[(对话上下文)]
        Router -->|分发任务| Agent1[客服智能体]
        Router -->|分发任务| Agent2[票务智能体]
        Router -->|分发任务| Agent3[文案/研学智能体]
    end
    
    subgraph 📚 知识与工具层 (本地私有化部署)
        Agent1 & Agent3 <-->|RAG 检索| Dify[本地 Dify 知识库集群]
        Agent2 <-->|API 执行| OpenClaw[OpenClaw 工具执行网关]
    end
    
    OpenClaw --> DB[(PostgreSQL 业务数据库)]
    OpenClaw --> WxPay[微信支付系统]
```

### 3.2 关键技术选型与解决思路

1. **中枢控制 (LangGraph)**：解决传统 Agent 容易“跑偏”死循环的问题。利用 LangGraph 的状态机（State Graph）特性，我们能够清晰地定义工作流，并在关键节点（如大额支付）加入 `Human-in-the-loop`（人工确认）机制。
2. **多智能体协作 (CrewAI)**：一个 AI 无法做好所有事。我们将系统拆分为“客服”、“票务”、“研学”等垂直 Agent。当用户问“门票多少钱，顺便帮我写个朋友圈”时，CrewAI 会将任务拆解，分别交由票务 Agent 和文案 Agent 并行处理。
3. **本地化部署隔离**：Dify 和 OpenClaw 通过 Docker 本地化部署，彻底阻断敏感业务数据流向公网，满足政企级项目的合规要求。

## 4. 🛠️ 核心开发链路与代码思想

### 4.1 智能体调用 Dify 知识库 (代码思路)

在 LangGraph 的节点中，我们通过标准 API 对接本地 Dify 服务：

```python
# 伪代码思路：调用本地 Dify 接口获取景区专业知识
def retrieve_scenic_knowledge(query: str) -> str:
    headers = {"Authorization": f"Bearer {LOCAL_DIFY_API_KEY}"}
    payload = {"inputs": {}, "query": query, "response_mode": "blocking", "user": "agent_01"}
    response = requests.post("http://localhost:5001/v1/chat-messages", json=payload, headers=headers)
    return response.json().get("answer", "")
```

### 4.2 票务智能体通过 OpenClaw 下单 (执行思路)

当识别到购票意图时，智能体构造参数，通过 OpenClaw 触发业务后端：

1. **意图提取**：提取出 `ticket_type="成人票"`, `count=2`, `date="2026-05-01"`。
2. **工具调用**：智能体自主决定调用 `create_ticket_order` 工具。
3. **闭环返回**：OpenClaw 返回订单号与支付二维码链接，智能体将其包装为自然语言返回给小程序端。
